# 如何完善企微 MCP Gateway 的可靠性、安全性与直连能力

本文定义企微数据中转 MCP Gateway 的加固设计。改动修复同步漏数和生产配置风险，增加可选企微直连模式，并建立可自动执行的测试基线。

## 文档计划

- **目标**：定义可直接转化为实施计划的组件边界、数据流、失败语义和验收标准
- **读者**：负责实现、审查、测试和部署该服务的开发者
- **内容**：现状证据、设计决策、组件职责、迁移策略、测试与验收
- **未决问题**：无。数据模式、存储边界和失败处理均已确认

## 当前问题与改进范围

本次改动采用“均衡加固”方案。它覆盖同步可靠性、生产安全、自动化测试、管理端敏感信息展示，以及可选的实时企微读取。

现有实现存在以下已验证问题：

- `app/wecom/dispatch.py` 最多处理 500 条详情，但仍把游标写到当前时间，剩余记录可能永久跳过
- 详情失败时，汇报和审批最小记录使用时间戳 `0`，按业务时间窗口查询时不可见
- `force=True` 会持久化回拨游标，与管理接口声明的语义不一致
- `app/crypto.py` 在缺少 `CREDENTIAL_KEY` 时使用仓库内固定密钥
- 生产环境示例没有设置 `WECOM_USE_MOCK=false`，而应用默认值为 `true`
- 管理端租户列表返回完整 MCP Token，任何列表读取都能批量获取凭证
- `tests/test_smoke_client.py` 在导入时运行网络请求，导致 pytest 在收集阶段失败
- SQL 初始化脚本、运行时数据定义和实际配置字段不一致

本次不引入消息队列、Redis 会话集群或新的数据库产品。智能表格读取、管理后台改版和无关模块重构不在范围内。

## 两种数据模式如何工作

每个租户通过 `tenant_config.data_mode` 选择数据来源。字段使用 `VARCHAR(16)`，允许值为 `stored` 和 `direct`，默认值为 `stored`。

| 模式 | MCP 读取来源 | 后台同步 | 业务表写入 | 中心配置与审计 |
|---|---|---|---|---|
| `stored` | 租户 MySQL 业务表 | 启用 | 启用 | 保留 |
| `direct` | 企业微信 API | 跳过 | 禁止 | 保留 |

`direct` 模式只禁止保存审批、汇报和打卡业务记录。租户配置、加密凭证、MCP Token 和审计日志继续写入 MySQL。切换模式不会删除历史业务数据。

所有 MCP 工具保持现有名称、参数和外层返回结构。返回字段 `source` 标记实际来源：`db`、`wecom` 或开发环境使用的 `mock`。直连失败时返回企微错误，不回退到历史缓存。

## 数据访问组件如何分流

新增 `app/data_access.py` 作为 MCP 工具与数据源之间的边界。该模块公开汇报、审批和打卡的列表与详情操作，并根据当前租户的 `data_mode` 选择数据库适配器或企微适配器。

组件职责如下：

- `app/auth.py`：把 `tenant_id`、`corpid`、运行时凭证、`schema_name` 和 `data_mode` 写入请求上下文
- `app/tenant.py`：从 `tenant_config` 加载并缓存 `data_mode`
- `app/data_access.py`：验证参数、选择数据源、规范返回结构并映射错误
- `app/mcp_server.py`：保留工具声明、审计和 JSON 序列化，不直接查询数据库或调用企微
- `app/wecom/*.py`：继续封装企微列表、详情、分页和时间分段逻辑
- `app/db.py`：继续负责租户业务表查询、幂等写入、游标和审计

该边界让两种数据模式共享 MCP 契约。实现可以独立测试数据分流，而不启动 MCP HTTP 服务。

## 直连模式如何返回数据

直连列表先调用企微列表接口，再为返回范围内的单号补取详情。`limit` 限制在 1 到 100，企微适配器最多补取 `limit` 条详情。

各工具按以下规则执行：

- `wecom_list_reports`：拉取汇报单号，补取详情，并映射为现有列表字段
- `wecom_get_report`：按 `journaluuid` 实时读取详情
- `wecom_list_approvals`：拉取审批单号，补取详情，并映射为现有列表字段
- `wecom_get_approval_detail`：按 `sp_no` 实时读取详情
- `wecom_list_checkins`：使用通讯录 Secret 自动获取 userid，或使用手工 userid 实时读取打卡记录
- `wecom_list_smart_table_records`：保持当前搁置状态

列表接口整体失败时，工具返回企微错误码、错误信息和 `source=wecom`。单条详情失败时，列表保留单号并添加 `_partial=true` 和错误摘要，同时返回 `partial_count`。打卡直连缺少可用 userid 时，工具返回可操作的配置错误。

直连调用记录工具名、目标、耗时和状态。审计失败不覆盖主调用结果，但应用必须记录审计写入异常。

## 缓存模式如何避免漏数

同步代码不再截取前 500 条后推进游标。每个来源窗口按以下顺序处理：

1. 拉取窗口内的全部唯一单号或打卡记录
2. 当数量超过 500 时，按时间中点拆分窗口并分别处理
3. 继续拆分，直到子窗口不超过 500 条或达到一分钟最小窗口
4. 最小窗口仍超过 500 条时，按 500 条一批处理该窗口的全部记录
5. 所有子窗口完成后，才把来源游标推进到本轮真实结束时间

数据库写入失败会阻止本轮游标提交。详情 API 失败但最小记录成功写入时，该单号视为已处理，并保留 `_partial` 标记供后续诊断和重试。

部分记录新增来源时间窗字段，不伪造业务提交时间。汇报和审批表保存 `source_window_start`、`source_window_end` 与 `is_partial`。查询使用真实业务时间；部分记录则按来源时间窗与请求窗口是否重叠来判断可见性。详情重试成功后，幂等更新会写入真实业务时间并清除 `is_partial`。

`force` 只让当前同步忽略已有游标并使用指定回看窗口。只有 `reset_cursor` 会先持久化回拨游标。管理接口、后台文案和测试必须使用同一语义。

## 生产配置如何拒绝危险默认值

应用在生产环境启动时执行配置验证。以下情况必须终止启动并给出具体变量名：

- `CREDENTIAL_KEY` 为空
- `ADMIN_PASSWORD` 为空或仍为示例值
- `WECOM_USE_MOCK=true`
- `DB_PASSWORD` 为空或仍为示例值

开发环境继续允许 mock 数据和本地凭证兜底。`.env.prod.example` 必须显式设置 `WECOM_USE_MOCK=false`，并说明直连与缓存模式都需要企微凭证。

数据库连接不再手工拼接密码。配置使用 SQLAlchemy URL 构造器，确保包含 `@`、`:`、`#` 或 `/` 的密码仍能正确解析。

## 管理端如何保护敏感信息

租户列表不返回完整 `mcp_token`。列表只返回 `has_mcp_token` 和末四位提示，防止一次请求批量暴露所有租户凭证。

新增经过管理员会话鉴权的 MCP 配置接口。用户点击 **MCP 配置** 后，前端调用该接口并显示完整连接 JSON。编辑租户时，Token 留空表示保持原值；新建租户仍要求 Token。

租户表格显示 **缓存模式** 或 **企微直连**。直连租户禁用 **同步**、**全量回拨** 和 **同步诊断**，并解释实时请求不会写入业务表。表单保存、同步和 Token 获取操作显示提交状态，防止重复请求。

## 数据库迁移如何保持兼容

新增 MySQL 5.7 兼容迁移脚本。脚本通过 `information_schema` 检查字段后执行变更，重复运行不会破坏数据。

迁移包括以下内容：

- `tenant_config.data_mode`，默认 `stored`
- 汇报和审批表的 `source_window_start`
- 汇报和审批表的 `source_window_end`
- 汇报和审批表的 `is_partial`
- 中心配置表当前由运行时补齐但 SQL 脚本缺失的字段
- 业务表当前由运行时创建但初始化脚本缺失的打卡结构

现有租户升级后保持 `stored`。新租户也默认使用 `stored`，除非管理员明确选择企微直连。迁移不删除 schema 或业务行。

## 测试如何证明设计有效

测试分为不依赖外部服务的单元测试和需要运行中服务的手动冒烟测试。pytest 不连接真实企微或 MySQL。

单元测试覆盖以下行为：

- `stored` 与 `direct` 选择正确适配器，并返回相同外层结构
- 直连模式不调用业务写入、游标写入或后台同步
- 汇报、审批和打卡直连的成功、整体失败和部分详情失败
- 超过 500 条时处理全部数据，并且只在成功后推进游标
- 数据库写入失败时不推进游标
- 部分记录在来源时间窗内可见，详情补全后使用真实业务时间
- `force` 不持久化回拨，`reset_cursor` 会持久化回拨
- 生产环境拒绝空密钥、示例密码和 mock 模式
- 租户列表不返回完整 Token，按需接口拒绝未认证请求
- 数据库密码包含 URL 特殊字符时仍能生成有效连接配置

`tests/test_smoke_client.py` 增加 `if __name__ == "__main__"` 入口保护，并允许通过环境变量配置服务地址和 Token。它保留为手动端到端检查，不参与 pytest 单元测试。

交付前执行以下验证：

```powershell
.\.venv\Scripts\python.exe -m compileall -q app
.\.venv\Scripts\python.exe -m pytest -q
pnpm --dir admin-ui build
docker compose config -q
```

前端构建允许保留现有大分块警告。本次不做前端代码分包，因为它不影响本轮可靠性与安全目标。

## 验收结果如何判定

实现满足以下条件后才能交付：

- 两种数据模式通过同一组 MCP 契约测试
- 直连模式没有审批、汇报或打卡业务写入
- 500 条以上的同步场景不漏数，也不错误推进游标
- 生产危险默认值无法启动应用
- 租户列表响应不含完整 MCP Token
- Python 编译、pytest、前端构建和 Compose 检查全部通过
- 代码审查没有未处理的 Critical 问题

Gemini 审查需要本机提供 `GEMINI_API_KEY`。如果凭证仍不可用，交付记录必须说明 Gemini 未完成审查，并保留 Claude 与本地验证结果。
