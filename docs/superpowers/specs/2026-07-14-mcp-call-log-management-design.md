# MCP 调用日志管理设计

## 1. 目标

为企微 MCP 网关增加统一的调用日志管理能力，覆盖：

- MCP 业务工具调用；
- MCP 协议访问；
- MCP 鉴权成功与失败；
- 管理后台的分页查询、结构化检索、统计概览和详情查看；
- 勾选、筛选条件、日期阈值及全部日志清理；
- 可配置的自动保留策略。

日志只用于运营排障和安全审计，不保存 MCP Token、企微 Secret、完整请求正文或完整响应正文。

## 2. 已确认范围

### 2.1 日志范围

日志分为三类：

- `tool`：业务工具执行，如 `wecom_list_reports`；
- `protocol`：MCP HTTP/JSON-RPC 协议访问，如 `initialize`、`tools/list`；
- `auth`：缺少 Token、无效 Token、鉴权通过等鉴权事件。

### 2.2 检索能力

管理页面支持以下结构化条件：

- 租户；
- 日志类别；
- 工具或事件；
- 状态；
- 时间范围；
- 关键词；
- 请求 ID；
- 客户端 IP；
- 最小和最大耗时。

不提供高级查询表达式，也不允许输入 SQL 条件。

### 2.3 清理能力

支持：

- 按所选日志 ID 清理；
- 按当前筛选结果清理；
- 清理指定日期之前的日志；
- 清空全部日志。

批量和全量操作必须先预览影响条数并二次确认。全部清空还需输入确认文字。

### 2.4 自动保留

- 默认保留 90 天；
- 管理员可设置 `0–3650` 天；
- `0` 表示关闭自动清理；
- 每天执行一次自动清理。

## 3. 架构决策

### 3.1 统一中心日志表

新增中心表 `mcp_call_log`。新产生的业务、协议和鉴权日志全部写入中心表。

现有各租户 schema 中的 `audit_log` 保留，以避免破坏历史 schema 结构，但新版本上线后停止写入。管理查询、统计和清理只访问中心表。

采用中心表的原因：

- 未识别租户的 401 没有可用的租户 schema；
- 全局分页和统计无需动态拼接多个 schema；
- 清理和保留策略只有一个数据源；
- 租户视角可通过 `tenant_id` 高效过滤。

### 3.2 统一审计模块

新增独立的 MCP 审计模块，负责：

- 构造标准日志事件；
- 参数和错误摘要脱敏；
- 字段长度限制；
- 容错写入；
- 查询条件规范化；
- 分批清理；
- 旧日志迁移。

日志写入失败不得改变 MCP 主请求的响应。

## 4. 数据模型

### 4.1 `mcp_call_log`

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT` | 自增主键 |
| `tenant_id` | `VARCHAR(64)` | 真实租户 ID；无法识别时为空字符串 |
| `category` | `VARCHAR(16)` | `tool/protocol/auth` |
| `event_name` | `VARCHAR(96)` | 工具名、JSON-RPC 方法或鉴权事件名 |
| `target` | `VARCHAR(256)` | 业务对象摘要 |
| `params_summary` | `VARCHAR(512)` | 脱敏参数摘要 |
| `result_status` | `VARCHAR(16)` | `ok/partial/error/denied` |
| `error_code` | `VARCHAR(64)` | HTTP 或安全业务错误码 |
| `error_summary` | `VARCHAR(256)` | 白名单安全错误摘要 |
| `cost_ms` | `INT` | 调用耗时，最小为 0 |
| `request_id` | `VARCHAR(64)` | 内部请求 ID 或安全的会话标识 |
| `client_ip` | `VARCHAR(64)` | 规范化客户端 IP |
| `http_method` | `VARCHAR(16)` | HTTP 方法 |
| `http_status` | `SMALLINT` | HTTP 状态码 |
| `legacy_schema` | `VARCHAR(64)` | 旧日志来源 schema；新日志为空 |
| `legacy_id` | `BIGINT` | 旧日志 ID；新日志为空 |
| `created_at` | `DATETIME(6)` | 事件发生时间 |

索引至少包括：

- `(tenant_id, created_at, id)`；
- `(created_at, id)`；
- `(category, event_name, created_at)`；
- `(result_status, created_at)`；
- `(request_id)`；
- `(client_ip, created_at)`；
- `(cost_ms, created_at)`；
- 唯一键 `(legacy_schema, legacy_id)`。

`legacy_schema` 与 `legacy_id` 对新日志均为 `NULL`，利用 MySQL 对含 `NULL` 唯一键的行为允许正常新日志重复为空。

### 4.2 `gateway_setting`

使用中心键值配置表保存运行时设置：

- `setting_key`：主键；
- `setting_value`：字符串值；
- `updated_at`：更新时间。

本功能使用键 `mcp_log_retention_days`。缺少该键时按 90 天处理。

## 5. 埋点与数据流

### 5.1 鉴权事件

`BearerTokenMiddleware` 在返回结果前记录：

- `auth_missing`：缺少 Bearer Token；
- `auth_invalid`：Token 无效；
- `auth_ok`：成功绑定租户上下文。

鉴权失败日志不保存 Token 原文、Token 摘要或可逆指纹。无法识别租户时 `tenant_id` 为空。

为防止恶意请求刷爆日志表，对相同“客户端 IP + 鉴权事件”限制单位时间内的写入数量。超过阈值后仅写应用告警，不再逐条写数据库。

### 5.2 协议事件

协议审计中间件记录：

- HTTP 方法与最终状态码；
- JSON-RPC 方法名；
- 请求 ID、客户端 IP、租户和耗时。

对于可解析且不超过安全大小限制的 JSON 请求，只提取 `method` 字段；请求正文解析后立即丢弃，不写入数据库。不可解析请求使用通用事件名 `mcp_http_request`。

中间件必须完整回放请求流，不能改变 FastMCP 的输入。

### 5.3 工具事件

现有 `_run_real` 迁移到统一审计模块：

- 正常结果写 `ok`；
- 存在部分失败写 `partial`；
- 安全业务错误和未预期异常写 `error`；
- Mock 路径同样记录 `tool` 日志。

工具层只保存明确构造的参数摘要，不对任意对象执行 `str()` 后直接入库。

## 6. 管理 API

所有接口均位于 `/admin` 下，并要求有效管理员会话。

### 6.1 查询

`GET /admin/mcp-logs`

查询参数：

- `tenant_id`、`category`、`event_name`、`status`；
- `from`、`to`；
- `q`；
- `request_id`、`client_ip`；
- `cost_min`、`cost_max`；
- `page`、`page_size`。

规则：

- 默认最近 24 小时；
- `page_size` 默认 20，最大 100；
- 关键词最大 100 字符；
- 所有 SQL 使用绑定参数；
- 返回 `items`、`total`、`page`、`page_size`。

### 6.2 统计

`GET /admin/mcp-logs/stats`

接受与列表相同的过滤条件，不接受分页参数，返回：

- 调用总数；
- 成功率；
- 错误数；
- 平均耗时；
- P95 耗时；
- 时间趋势；
- 工具排行；
- 状态分布。

统计时间范围与页面时间筛选保持一致。MySQL 5.7 没有窗口百分位函数，P95 使用有界时间范围内的排序位置查询计算。

### 6.3 清理预览

`POST /admin/mcp-logs/delete-preview`

请求体支持模式：

- `ids`；
- `filter`；
- `before_date`；
- `all`。

返回：

- 匹配条数；
- 预览时的最大日志 ID；
- 5 分钟有效的签名确认令牌。

确认令牌绑定：清理模式、规范化条件、最大日志 ID、匹配条数、管理员会话和过期时间。

### 6.4 执行清理

`DELETE /admin/mcp-logs`

请求必须与预览条件一致并携带确认令牌。执行时只删除 `id <= preview_max_id` 的匹配记录，避免预览后新写入的日志被误删。

删除以每批最多 5000 条执行，中断后可安全重试。响应返回实际删除条数。

### 6.5 保留设置

- `GET /admin/mcp-log-settings`；
- `PUT /admin/mcp-log-settings`。

只接受整数 `0–3650`。

## 7. 自动清理

现有 APScheduler 增加每日任务：

1. 读取 `mcp_log_retention_days`；
2. 值为 0 时跳过；
3. 计算截止时间；
4. 按 5000 条循环删除；
5. 每批独立提交；
6. 使用安全应用日志记录删除条数与耗时。

自动任务与手动清理复用同一个分批删除服务。

## 8. 旧日志迁移

启动迁移在建好中心表后执行：

1. 从 `tenant_config` 获取真实租户 ID 与 schema 映射；
2. 枚举合法租户 schema；
3. 读取最近 90 天的旧 `audit_log`；
4. 转为 `category=tool` 的中心日志；
5. 写入 `legacy_schema` 和 `legacy_id`；
6. 使用唯一键忽略重复数据。

迁移允许重复执行。单个租户迁移失败不删除旧数据，但启动日志必须明确指出失败租户；中心表创建失败则阻止应用启动。

## 9. 管理页面

### 9.1 信息架构

顶部导航增加：

- 租户管理；
- 调用日志。

全局与租户日志共用统一日志工作台。租户列表行菜单新增“查看调用日志”，跳转到调用日志页面并通过 URL 设置 `tenant_id`。

清除租户条件即可返回全局视角。筛选状态同步 URL，可刷新恢复和复制分享。

### 9.2 租户仪表盘

当选择单个租户时，页面顶部展示：

- 调用总数；
- 成功率；
- 错误数；
- 平均和 P95 耗时；
- 调用趋势；
- 工具排行；
- 状态分布。

仪表盘随时间与其他筛选条件联动。全局视角以日志检索为主，不展示租户仪表盘。

### 9.3 筛选与表格

基础筛选显示租户、类别、状态、时间和关键词；请求 ID、IP、事件与耗时区间放入“更多筛选”。

表格列：

- 时间；
- 租户；
- 类别；
- 事件；
- 状态；
- 耗时；
- 客户端 IP；
- 摘要。

点击行打开右侧详情抽屉。抽屉只展示 API 返回的白名单字段。

### 9.4 清理与设置

- 勾选日志后出现批量操作栏；
- 条件清理、日期清理和全部清空放入危险操作菜单；
- 清理预览对话框展示影响条数；
- 全部清空要求输入指定确认文字；
- 页面右上角通过“日志设置”抽屉配置保留天数。

### 9.5 响应式与状态

- 窄屏隐藏租户、IP、摘要等次要列；
- 详情抽屉保留完整信息；
- 分别提供加载、空数据、加载错误、统计错误和重试状态；
- 查询失败保留当前筛选条件。

## 10. 安全约束

- 禁止保存 Authorization、Token、Secret、Cookie 和完整请求/响应正文；
- 参数摘要只允许由各事件显式构造；
- 错误摘要使用白名单映射，不保存原始异常信息；
- 输出前再次脱敏；
- 客户端 IP 必须规范化；
- 所有筛选和清理条件使用绑定参数；
- 清理确认令牌有签名、会话绑定和短期过期时间；
- 全部清空不能通过单次无预览请求完成。

## 11. 异常处理

- 审计写入失败不影响 MCP 主流程；
- 管理 API 不暴露 SQL、schema 或异常堆栈；
- 统计失败不阻断日志列表，前端单独展示统计错误；
- 自动清理失败保留未完成数据，下次任务继续；
- 旧日志迁移不修改或删除源表数据。

## 12. 测试与验收

### 12.1 后端

- 三类日志写入与状态映射；
- Mock 工具路径日志；
- 敏感信息不入库、不出现在响应和应用日志；
- 鉴权日志限速；
- 协议请求流完整回放；
- 组合筛选、分页、时间默认值和参数边界；
- 总量、成功率、错误数、平均值、P95、趋势和排行；
- 迁移最近 90 天且重复执行不重复；
- 四种清理模式、预览令牌、防新增误删和分批删除；
- 保留天数 `0/1/90/3650` 边界；
- 自动清理关闭、成功、中断重试；
- 所有管理接口的未登录访问返回 401。

### 12.2 前端

- 顶部导航切换；
- URL 筛选恢复；
- 租户快捷入口；
- 仪表盘随筛选刷新；
- 表格分页、选择和详情抽屉；
- 四种清理入口和确认文案；
- 保留天数设置；
- 加载、空数据、错误和窄屏状态。

### 12.3 最终验证

- 完整 Python 测试；
- 前端单元测试；
- 前端生产构建；
- Ant Design 静态检查；
- 浏览器桌面与窄屏交互验证；
- 数据库迁移在空库和已有租户数据环境中验证。

## 13. 非目标

本次不实现：

- 高级查询表达式；
- 原始 SQL 查询；
- 完整请求/响应正文留存；
- 日志导出；
- 外部日志系统或 SIEM 集成；
- 分区表和冷热归档。
