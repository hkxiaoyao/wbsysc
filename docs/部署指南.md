# 生产部署指南

> 阶段三交付：Docker + Nginx/HTTPS + systemd + 部署文档。本机 Docker Desktop 未运行，镜像构建需主人启动后执行（命令已附）。

## 一、部署形态

```
Internet ──HTTPS──▶ Nginx(443) ──▶ uvicorn(:8001, 裸机或容器)
                        │              ├─ MCP /mcp (Bearer Token 鉴权)
                        │              ├─ 管理 /admin/* (session 鉴权)
                        │              └─ /admin/ui (静态)
                        ▼
                   MySQL(分schema) + 可选 Redis
```

两种部署方式任选其一：
- **A. Docker Compose**（推荐，自带构建+健康检查+日志轮转）
- **B. 裸机 systemd**（已有云主机 + Nginx 时）

### 数据读取模式

每个连接实例可选择 `stored` 或 `direct`。`stored` 定时把业务数据同步到该连接的 MySQL schema，MCP 查询本地表。`direct` 在每次 MCP 调用时请求企微 API，不写入审批、汇报或打卡业务表，也不参加后台同步。

两种模式都在 MySQL 保存连接配置、加密凭证、连接 MCP Token 和审计日志。直连请求失败时会返回企微错误，不读取历史缓存。旧租户企业微信配置会兼容回填为默认连接并保持 `stored`。

直连模式查询大时间窗口时，会从最新时间分段开始遍历企微列表分页，并为返回的单号逐条请求详情，API 调用成本较高。生产调用建议缩小时间窗口并设置较小的 `limit`。

### 生产启动要求

生产环境必须设置 `WECOM_USE_MOCK=false`，并为 `CREDENTIAL_KEY`、`MCP_TOKEN_HMAC_KEY`、`MCP_TOKEN_PLAINTEXT_KEY`、`ADMIN_PASSWORD` 和 `DB_PASSWORD` 填写非空、非示例的安全值；三个密钥均至少 32 个 UTF-8 字节且两两不同。任一项不符合要求时应用会拒绝启动。首次升级保持 `MCP_SERVICE_ENABLED=false`。

## 二、A 方式：Docker Compose

### 1. 准备配置
```bash
cp .env.prod.example .env
# 确认 WECOM_USE_MOCK=false、MCP_SERVICE_ENABLED=false，并填写密码及三个独立密钥
# 每个密钥分别生成一次；不要复用输出：
python -c "import secrets; print(secrets.token_urlsafe(48))"
```

### 2. 构建 + 启动

生产服务器推荐直接运行 `bash deploy/server_deploy.sh`。脚本会在拉取镜像和启动应用前校验生产配置，并自动使用宿主 `mysql` CLI 严格执行 `004` → `005` → `006` → `007` → `008` → `009` → `010`。它先强制关闭服务功能并重建、验证健康状态，再按发布前请求值选择是否启用和二次重建。

```bash
docker compose build          # 多阶段：node 构前端 → python 运行
docker compose up -d
docker compose logs -f
```

### 3. 验证
```bash
curl http://127.0.0.1:8001/health            # 仅宿主机本地检查；对外统一使用 HTTPS
docker compose ps                            # 状态
# Nginx 反代后：curl https://mcp.example.com/health
```

容器默认只信任 `172.16.0.0/12` 内的 Docker 代理来源。若宿主机通过
`default-address-pools` 或自定义网络将 Docker 网关分配到其他网段，需要同步调整
Dockerfile 中的 `--forwarded-allow-ips`，否则 HTTPS 写请求会因无法识别原始
scheme 而被同源校验拒绝。

### 4. 接入首个租户

1. 平台管理员在管理后台创建租户 ID、名称与初始密码。
2. 租户使用 ID/密码登录租户后台。
3. 平台管理员或租户创建企业微信连接实例，在连接中配置 CorpID、凭据、数据模式、同步策略与可信域名，并可上传企业微信校验文件。

### 5. 升级

升级顺序必须是：备份数据库、执行 `004` → `005` → `006` → `007` → `008` → `009` → `010`、以功能关闭状态重建并健康检查，再按批准决定是否启用。`008` 依赖 `005` 和 `006`，不可跳序。推荐让部署脚本自动完成：

```bash
git pull
bash deploy/server_deploy.sh
```

脚本从 `.env` 读取非敏感数据库参数；迁移账户和密码只从发布终端环境读取且必须不同于运行账户。迁移主机默认使用宿主 `127.0.0.1`，可覆盖。Compose 本身没有 MySQL 服务，因此宿主必须先安装 `mysql` CLI；脚本通过 `MYSQL_PWD` 传递密码，不会把密码写入命令行或日志。

手动升级时，先确认下列非敏感连接参数与 `.env` 一致，再执行迁移；只有命令成功后才能拉取镜像并启动：

```bash
DB_MIGRATION_HOST=127.0.0.1
DB_PORT=3306
DB_MIGRATION_USER=wbsysc_migrator
DB_NAME=websysc
read -rsp "DB_MIGRATION_PASSWORD: " MYSQL_PWD && export MYSQL_PWD && echo
mysql --protocol=TCP --host="$DB_MIGRATION_HOST" --port="$DB_PORT" \
  --user="$DB_MIGRATION_USER" "$DB_NAME" < sql/004_gateway_hardening.sql
mysql --protocol=TCP --host="$DB_MIGRATION_HOST" --port="$DB_PORT" --user="$DB_MIGRATION_USER" "$DB_NAME" < sql/005_mcp_call_log.sql
mysql --protocol=TCP --host="$DB_MIGRATION_HOST" --port="$DB_PORT" --user="$DB_MIGRATION_USER" "$DB_NAME" < sql/006_connection_platform.sql
mysql --protocol=TCP --host="$DB_MIGRATION_HOST" --port="$DB_PORT" --user="$DB_MIGRATION_USER" "$DB_NAME" < sql/007_tenant_auth.sql
mysql --protocol=TCP --host="$DB_MIGRATION_HOST" --port="$DB_PORT" --user="$DB_MIGRATION_USER" "$DB_NAME" < sql/008_mcp_service.sql
mysql --protocol=TCP --host="$DB_MIGRATION_HOST" --port="$DB_PORT" --user="$DB_MIGRATION_USER" "$DB_NAME" < sql/009_tenant_identity_boundary.sql
mysql --protocol=TCP --host="$DB_MIGRATION_HOST" --port="$DB_PORT" --user="$DB_MIGRATION_USER" "$DB_NAME" < sql/010_connection_domain_verify.sql
unset MYSQL_PWD
docker pull ghcr.io/hkxiaoyao/wbsysc:latest
sed -i 's/^MCP_SERVICE_ENABLED=.*/MCP_SERVICE_ENABLED=false/' .env
docker compose up -d
# 验证 /health 的 mcp_service_enabled=false；批准启用后改 true 并 docker compose up -d --force-recreate
```

> `004_gateway_hardening.sql` 包含 `DELIMITER` 和存储过程语句，必须使用 MySQL `mysql` CLI 执行；迁移失败应立即终止升级。不要因功能回滚删除 `008` 表或数据。

## 三、B 方式：裸机 systemd

### 1. 环境
```bash
useradd -r wbsysc
mkdir -p /opt/wbsysc && chown wbsysc:wbsysc /opt/wbsysc
# 拷代码到 /opt/wbsysc
sudo -u wbsysc python3.11 -m venv /opt/wbsysc/.venv
sudo -u wbsysc /opt/wbsysc/.venv/bin/pip install -r requirements.txt
# 前端构建
cd admin-ui && pnpm install && pnpm build   # 产出 app/static/dist
cd ..
```

### 2. 配置
```bash
cp .env.prod.example /opt/wbsysc/.env
# 编辑填入真实值（DB/ADMIN_PASSWORD/三个独立密钥），首次保持 MCP_SERVICE_ENABLED=false
# 严格权限
chmod 600 /opt/wbsysc/.env
chown wbsysc:wbsysc /opt/wbsysc/.env
```

### 3. systemd 服务
```bash
cp deploy/wbsysc.service /etc/systemd/system/
systemctl daemon-reload
systemctl enable --now wbsysc
systemctl status wbsysc
journalctl -u wbsysc -f      # 看日志
```

## 四、Nginx + HTTPS

### 1. 装证书
```bash
mkdir -p /etc/nginx/ssl
# 放 fullchain.pem + privkey.pem（Let's Encrypt 或购买的证书）
chmod 600 /etc/nginx/ssl/privkey.pem
```

### 2. 配置
```bash
cp deploy/nginx.conf /etc/nginx/conf.d/wbsysc.conf
# 编辑：server_name 改成真实域名；证书路径核对
nginx -t && systemctl reload nginx
```

### 3. 关键点（MCP 透传）
Nginx 配置已处理 MCP Streamable HTTP 的三个要点：
- `proxy_buffering off` —— SSE 流式响应必须关缓冲
- `proxy_read_timeout 600s` —— 长会话不超时
- 支持 POST/DELETE —— MCP transport 会用 DELETE 关会话

## 五、安全检查清单（上线前必过）

- [ ] `.env` 权限 600，owner=wbsysc
- [ ] `ADMIN_PASSWORD` 改强密码（非 admin123）
- [ ] `CREDENTIAL_KEY` 配强随机 48 字节串
- [ ] `MCP_TOKEN_HMAC_KEY`、`MCP_TOKEN_PLAINTEXT_KEY` 各自为强随机值，且三个密钥两两不同
- [ ] `MCP_SERVICE_ENABLED=false` 的第一阶段健康检查通过；启用需审批并重建
- [ ] DB 账户受限（仅 DDL/DML on wbd_* + tenant_config，无 GRANT）
- [ ] DB 密码改强密码（非 websysc）
- [ ] `WECOM_USE_MOCK=false`，禁止生产环境使用 mock 数据
- [ ] 企微可信 IP 白名单加入**服务器公网出口 IP**（非开发机 IP）
- [ ] HTTPS 证书有效，HTTP 80 强制跳 443
- [ ] 各客户租户 secret 重新签发（开发期明文出现的已作废）
- [ ] 客户书面授权完成（见 plan 第 8 节红线）

## 六、运维

### 服务功能回滚

把 `MCP_SERVICE_ENABLED=false` 写回 `.env`，执行 `docker compose up -d --force-recreate`（或 systemd 重启），并确认 `/health` 返回 `mcp_service_enabled:false`。服务 `/mcp/service/{service_id}`、租户服务管理 UI/API 和服务运行时会关闭；旧 `/mcp`、`/mcp/{connection_id}` 与连接 Token 继续可用；平台管理员仍可通过认证后的管理 API 撤销服务 Token、禁用服务和精确清理。保留 `008` 结构和数据。启用阶段失败时发布脚本会自动恢复 `false`、重建、验证关闭态并以非零退出。

默认服务只会在功能为 `true` 的重启中、可信连接器注册后做幂等水位回填；它创建工具绑定但绝不复制连接 Token。

### 密钥与租户登录

`CREDENTIAL_KEY` 轮换前重加密全部凭证。当前 `MCP_TOKEN_HMAC_KEY` 是单 key 验证，旧 key 下预签发的 Token 切换后也会失效：先盘点全部旧 token ID，在维护窗口切换 key 并重启，再在新 key 下逐一签发、分发、验证并核对旧 ID 全部失效/撤销；窗口内会短暂中断。`MCP_TOKEN_PLAINTEXT_KEY` 轮换前重加密全部未撤销服务 Token 密文（已撤销密文已清空）。不得直接替换任一密钥。

连接 Token 仅签发时显示一次且不可揭示；未撤销服务 Token 仅当前平台管理员或所属租户可通过限流、审计、`no-store` 的端点揭示。管理员创建租户时必须设置初始密码，也可重置密码或修改登录状态；后台从不读取旧密码。重置/自行改密或禁用登录会撤销该租户会话，用户必须重新登录。

| 操作 | 命令 |
|------|------|
| 看日志 | `journalctl -u wbsysc -f` 或 `docker compose logs -f` |
| 重启 | `systemctl restart wbsysc` / `docker compose restart` |
| 手动同步某连接 | 连接实例工作台「立即同步」，或连接级 `sync` API |
| 新增租户 | 管理后台「新增租户」，填写 ID、名称和初始密码 |
| 新增企业微信连接 | 平台或租户后台「连接实例」，填写 CorpID、凭据、数据模式、同步策略和可信域名；保存域名后上传校验文件并复制公网 URL |
| 查同步游标 | `SELECT * FROM wbd_xxx.sync_cursor` |
| 死信排查 | 查 `wbd_xxx.audit_log` result_status=error 的记录 |

## 七、监控（可选，后续接入）

- `/health` 接健康探针（已配 Docker HEALTHCHECK + Nginx /health）
- 同步任务日志含 `同步轮次开始/结束` + per-tenant `pulled/stored/err`，可接日志平台告警
- err 持续 >0 或 `error` 字段出现 → 告警（建议接腾讯云监控 CLS）
