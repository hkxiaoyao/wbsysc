# 生产部署指南

> 阶段三交付：Docker + Nginx/HTTPS + systemd + 部署文档。本机 Docker Desktop 未运行，镜像构建需主人启动后执行（命令已附）。

## 一、部署形态

```
Internet ──HTTPS──▶ Nginx(443) ──▶ uvicorn(:8001, 裸机或容器)
                        │              ├─ MCP /mcp (Bearer Token 鉴权)
                        │              ├─ 管理 /admin/* (session 鉴权)
                        │              └─ /admin/ui (静态)
                        ▼
                   MySQL(分schema) + 可选 Redis
```

两种部署方式任选其一：
- **A. Docker Compose**（推荐，自带构建+健康检查+日志轮转）
- **B. 裸机 systemd**（已有云主机 + Nginx 时）

### 数据读取模式

每个租户可选择 `stored` 或 `direct`。`stored` 定时把业务数据同步到租户 MySQL schema，MCP 查询本地表。`direct` 在每次 MCP 调用时请求企微 API，不写入审批、汇报或打卡业务表，也不参加后台同步。

两种模式都在 MySQL 保存租户配置、加密凭证、MCP Token 和审计日志。直连请求失败时会返回企微错误，不读取历史缓存。现有租户升级后保持 `stored`。

直连模式查询大时间窗口时，会从最新时间分段开始遍历企微列表分页，并为返回的单号逐条请求详情，API 调用成本较高。生产调用建议缩小时间窗口并设置较小的 `limit`。

### 生产启动要求

生产环境必须设置 `WECOM_USE_MOCK=false`，并为 `CREDENTIAL_KEY`、`ADMIN_PASSWORD` 和 `DB_PASSWORD` 填写非空、非示例的安全值；任一项不符合要求时应用会拒绝启动。`stored` 与 `direct` 都需要这些中心配置，其中企微凭证通过管理后台或 `tenant_init` 加密写入 MySQL。

## 二、A 方式：Docker Compose

### 1. 准备配置
```bash
cp .env.prod.example .env
# 确认 WECOM_USE_MOCK=false，填入 DB_PASSWORD / ADMIN_PASSWORD / CREDENTIAL_KEY（生成见下）
# 生成 CREDENTIAL_KEY:
python -c "import secrets;print(secrets.token_urlsafe(48))"
```

### 2. 构建 + 启动

生产服务器推荐直接运行 `bash deploy/server_deploy.sh`。脚本会在拉取镜像和启动应用前校验生产配置，并自动使用宿主 `mysql` CLI 执行 `sql/004_gateway_hardening.sql`。

```bash
docker compose build          # 多阶段：node 构前端 → python 运行
docker compose up -d
docker compose logs -f
```

### 3. 验证
```bash
curl http://127.0.0.1:8001/health            # 容器健康
docker compose ps                            # 状态
# Nginx 反代后：curl https://mcp.example.com/health
```

### 4. 接入首个租户
```bash
# 在容器内跑 tenant_init（或用管理后台 UI）
docker compose exec wbsysc python -m app.tenant_init \
  --tenant-id customerA --corpid wwXXX --secret XXX \
  --token $(openssl rand -hex 24) \
  --contact-secret XXXX \
  --modules report,approval,checkin --display "客户A"
```

### 5. 升级

升级顺序必须是：备份数据库、执行 `sql/004_gateway_hardening.sql`、拉取或构建新镜像、启动新应用。推荐让部署脚本自动完成配置校验、迁移和切换：

```bash
git pull
bash deploy/server_deploy.sh
```

脚本从 `.env` 读取 `DB_PORT`、`DB_USER`、`DB_NAME` 和 `DB_PASSWORD`，迁移主机默认使用宿主 `127.0.0.1`，可通过执行环境或 `.env` 中的 `DB_MIGRATION_HOST` 覆盖。Compose 本身没有 MySQL 服务，因此宿主必须先安装 `mysql` CLI；脚本通过 `MYSQL_PWD` 传递密码，不会把密码写入命令行或日志。

手动升级时，先确认下列非敏感连接参数与 `.env` 一致，再执行迁移；只有命令成功后才能拉取镜像并启动：

```bash
DB_MIGRATION_HOST=127.0.0.1
DB_PORT=3306
DB_USER=wbsysc_app
DB_NAME=websysc
read -rsp "DB_PASSWORD: " MYSQL_PWD && export MYSQL_PWD && echo
mysql --protocol=TCP --host="$DB_MIGRATION_HOST" --port="$DB_PORT" \
  --user="$DB_USER" "$DB_NAME" < sql/004_gateway_hardening.sql
unset MYSQL_PWD
docker pull ghcr.io/hkxiaoyao/wbsysc:latest
docker compose up -d
```

> `004_gateway_hardening.sql` 包含 `DELIMITER` 和存储过程语句，必须使用 MySQL `mysql` CLI 执行；迁移失败应立即终止升级，不得切换到新应用。

## 三、B 方式：裸机 systemd

### 1. 环境
```bash
useradd -r wbsysc
mkdir -p /opt/wbsysc && chown wbsysc:wbsysc /opt/wbsysc
# 拷代码到 /opt/wbsysc
sudo -u wbsysc python3.11 -m venv /opt/wbsysc/.venv
sudo -u wbsysc /opt/wbsysc/.venv/bin/pip install -r requirements.txt
# 前端构建
cd admin-ui && pnpm install && pnpm build   # 产出 app/static/dist
cd ..
```

### 2. 配置
```bash
cp .env.prod.example /opt/wbsysc/.env
# 编辑填入真实值（DB/ADMIN_PASSWORD/CREDENTIAL_KEY）
# 严格权限
chmod 600 /opt/wbsysc/.env
chown wbsysc:wbsysc /opt/wbsysc/.env
```

### 3. systemd 服务
```bash
cp deploy/wbsysc.service /etc/systemd/system/
systemctl daemon-reload
systemctl enable --now wbsysc
systemctl status wbsysc
journalctl -u wbsysc -f      # 看日志
```

## 四、Nginx + HTTPS

### 1. 装证书
```bash
mkdir -p /etc/nginx/ssl
# 放 fullchain.pem + privkey.pem（Let's Encrypt 或购买的证书）
chmod 600 /etc/nginx/ssl/privkey.pem
```

### 2. 配置
```bash
cp deploy/nginx.conf /etc/nginx/conf.d/wbsysc.conf
# 编辑：server_name 改成真实域名；证书路径核对
nginx -t && systemctl reload nginx
```

### 3. 关键点（MCP 透传）
Nginx 配置已处理 MCP Streamable HTTP 的三个要点：
- `proxy_buffering off` —— SSE 流式响应必须关缓冲
- `proxy_read_timeout 600s` —— 长会话不超时
- 支持 POST/DELETE —— MCP transport 会用 DELETE 关会话

## 五、安全检查清单（上线前必过）

- [ ] `.env` 权限 600，owner=wbsysc
- [ ] `ADMIN_PASSWORD` 改强密码（非 admin123）
- [ ] `CREDENTIAL_KEY` 配强随机 48 字节串
- [ ] DB 账户受限（仅 DDL/DML on wbd_* + tenant_config，无 GRANT）
- [ ] DB 密码改强密码（非 websysc）
- [ ] `WECOM_USE_MOCK=false`，禁止生产环境使用 mock 数据
- [ ] 企微可信 IP 白名单加入**服务器公网出口 IP**（非开发机 IP）
- [ ] HTTPS 证书有效，HTTP 80 强制跳 443
- [ ] 各客户租户 secret 重新签发（开发期明文出现的已作废）
- [ ] 客户书面授权完成（见 plan 第 8 节红线）

## 六、运维

| 操作 | 命令 |
|------|------|
| 看日志 | `journalctl -u wbsysc -f` 或 `docker compose logs -f` |
| 重启 | `systemctl restart wbsysc` / `docker compose restart` |
| 手动同步某租户 | 管理后台「同步」按钮，或 `POST /admin/tenants/{id}/sync` |
| 新增租户 | 管理后台「新增租户」，自动建 schema |
| 查同步游标 | `SELECT * FROM wbd_xxx.sync_cursor` |
| 死信排查 | 查 `wbd_xxx.audit_log` result_status=error 的记录 |

## 七、监控（可选，后续接入）

- `/health` 接健康探针（已配 Docker HEALTHCHECK + Nginx /health）
- 同步任务日志含 `同步轮次开始/结束` + per-tenant `pulled/stored/err`，可接日志平台告警
- err 持续 >0 或 `error` 字段出现 → 告警（建议接腾讯云监控 CLS）
