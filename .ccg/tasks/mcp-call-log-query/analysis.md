# MCP 调用日志架构分析

## 现状

- 每个租户 schema 已有 `audit_log`，仅记录真实业务工具调用。
- 现有 `audit_log.tenant_id` 实际写入 schema 名，而不是真实租户 ID。
- Mock、协议访问、缺失或无效 Token 的 401 不记录。
- 跨租户分页、统计、清理需要动态拼接多个 schema。

## 方案

### 方案 A：中心日志表（推荐）

- 中心库新增统一事件表，记录 tool/protocol/auth 三类日志。
- 新流量只写中心表，旧租户日志表保留兼容，不再继续写入。
- 全局和租户视图均通过中心表过滤，统计与清理简单可靠。

### 方案 B：继续使用分租户日志表

- 扩展每个租户的 `audit_log`，跨租户查询时动态 `UNION ALL`。
- 未识别租户的鉴权失败还需额外系统日志表。
- 隔离性强，但形成双查询源，分页、统计、清理成本较高。

### 方案 C：中心表与租户表双写

- 新日志同时写中心表和租户表。
- 查询方便且保留租户副本，但会带来一致性、清理和容量双倍成本。

## 建议

采用方案 A。中心日志是运营与审计元数据，不是租户业务数据；统一存储更适合未识别租户的鉴权事件、全局统计、结构化筛选和受控清理。
