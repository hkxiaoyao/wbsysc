# 项目细节完善需求

## 目标

在不进行架构级重写的前提下，提高企微数据中转 MCP Gateway 的同步可靠性、生产安全性和可回归验证能力，并修复少量直接影响运维安全与使用体验的管理端问题。

## 已确认范围

1. 同步可靠性
   - 单次结果超过处理上限时不得静默推进游标并漏掉剩余数据。
   - 详情获取失败后的最小记录仍应能在对应同步时间窗内被查询。
   - `force` 与 `reset_cursor` 的行为和 API 文档保持一致。
2. 生产安全
   - 生产环境不得因缺省配置静默使用固定凭证加密密钥。
   - 生产配置必须明确关闭企微 mock 模式。
   - 管理端租户列表不得默认暴露完整 MCP Token。
3. 自动化验证
   - 将当前导入即执行、依赖外部服务的冒烟脚本与 pytest 单元测试隔离。
   - 为同步游标、部分记录、安全配置与鉴权边界补充自动化测试。
4. 细节完善
   - 对齐 SQL 初始化脚本、环境模板和实际运行时结构。
   - 修复与上述能力直接相关的管理端提示、加载和敏感信息展示细节。
5. 可配置的数据读取模式
   - 每个租户可统一选择 `stored` 或 `direct` 模式。
   - `stored` 模式继续定时同步业务数据到租户 MySQL schema，MCP 从本地业务表查询。
   - `direct` 模式下，每次 MCP 调用实时请求企微 API，不保存审批、汇报、打卡业务记录，也不运行该租户的同步任务。
   - 两种模式均继续使用中心 MySQL 保存租户配置、加密凭证、MCP Token 和审计日志。
   - 直连调用失败时返回明确错误，不静默回退到本地缓存。

## 成功标准

- Python 编译、后端测试、前端生产构建和 Compose 配置检查全部通过。
- 超过单轮处理上限、详情失败、强制窗口和游标回拨均有可重复的自动化测试。
- `APP_ENV=prod` 下缺少关键安全配置会明确失败，不会静默降级。
- 管理列表响应不再包含可直接使用的完整 MCP Token；必要的复制流程仍可受控完成。
- 同一套 MCP 工具在 `stored` 与 `direct` 两种模式下保持一致的入参和返回结构。
- 文档和示例配置与实际默认行为一致。

## 非目标

- 不引入消息队列、Redis 会话集群或新的数据库架构。
- 不实现一期已搁置的智能表格能力。
- 不重做管理后台视觉设计。
- 不进行与本次可靠性、安全性无关的大规模模块拆分。
