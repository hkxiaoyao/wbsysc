# 架构评估：连接实例与 MCP 服务融合

## 结论

如果产品约束确定为“一个连接实例只对应一个 MCP 暴露入口”，则应把 MCP 作为连接实例的内建能力，不再保留用户可见且与连接实例 1:1 的独立 MCP Service 聚合。

当前代码已经具备目标模型的主要能力：

- 连接级 MCP 地址：`/mcp/{connection_id}`
- 连接级令牌
- 连接级工具启停与策略
- 连接级调用日志

当前独立 MCP Service 的主要独有价值，是把多个连接实例的工具组合到一个 MCP 地址，并提供独立服务令牌、服务生命周期、服务级工具别名和策略。若产品不需要这种跨连接聚合，它就是重复抽象。

## 推荐目标模型

- Tenant 拥有多个 Connection Instance。
- 每个 Connection Instance 固定拥有一个 MCP Endpoint，MCP 不再是独立实体。
- 连接实例负责连接凭据、数据/同步策略、可信域名、附件验证等连接器配置。
- MCP 令牌、工具启停/策略、别名和调用日志都归属 Connection Instance。
- 对外统一使用 `/mcp/{connection_id}`。
- 移除独立“服务”菜单、服务创建流程和 `/mcp/service/{service_id}` 的新使用入口。

## 会失去的能力

- 将多个连接实例的工具聚合为一个 MCP 地址。
- 服务与连接实例分别启停、分别发令牌。
- 同一个连接工具在多个服务里配置不同别名或策略。
- 独立的服务版本与生命周期。

如果未来确实需要聚合，应将其定义为可选的“组合 MCP/发布端点”，而不是为每个连接自动创建一个 1:1 影子服务。

## 实施前置条件

必须先盘点生产数据中是否存在一个服务绑定多个连接的记录。若存在，需要产品决定拆分成多个连接级 MCP，或保留可选聚合模型，不能直接删除。

## 渐进迁移

1. 停止创建新的独立 MCP Service，前端以连接实例为唯一入口。
2. 将默认服务的令牌、工具策略、别名和状态迁移或映射到连接实例。
3. 为旧 `/mcp/service/{service_id}` 提供临时兼容映射，并记录调用量。
4. 调用日志以 `connection_id` 为主，逐步废弃 `service_id`。
5. 兼容期结束且确认无跨连接聚合后，删除服务路由、页面、网关与数据表。

## 术语影响

`CONTEXT.md` 当前把 Connection Instance 与 MCP Service 定义为两个聚合边界。只有产品正式确认取消跨连接聚合后，才应修改该术语定义；本次评估不直接改动领域词汇。
